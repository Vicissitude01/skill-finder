# 报告模板参考

本文件是 SKILL.md 中报告结构的补充说明，提供各字段的详细编写指南。

## 报告结构（八节）

1. **搜索信息头** — 需求原文、技术翻译、关键词、时间、数据来源、候选数
2. **加权排名表** — 按综合分降序，含匹配度/活跃度/综合分/象限列
3. **双轴象限图** — ASCII 绘制，四象限标注项目位置
4. **逐个详评** — 每项目：介绍 + 优点 + 缺点 + 用法
5. **功能矩阵** — ✅/❌/⚠️ 对比表格
6. **场景推荐** — 🏆最佳 + 💎冷门精品 + 场景表格
7. **最终建议** — 2-3句总结

## 字段编写指南

### 加权排名表

| 列 | 说明 |
|----|------|
| # | 排名序号 |
| 项目 | `[name](url)` 格式 |
| ⭐ Stars | 原始星标数 |
| 匹配度 | 1-5 整数 |
| 活跃度 | 1-5 整数 |
| 综合分 | 保留两位小数，公式：匹配度×0.6 + 星标分×0.25 + 活跃度×0.15 |
| 象限 | ⭐热门精准 / 🔥热门但偏 / 💎冷门精品 / 淘汰 |

### 象限判定

| 条件 | 象限标签 |
|------|---------|
| 匹配度 ≥ 3.5 且 星标分 ≥ 3 | ⭐热门精准 |
| 匹配度 ≥ 3.5 且 星标分 < 3 | 💎冷门精品 |
| 匹配度 < 3.5 且 星标分 ≥ 3 | 🔥热门但偏 |
| 匹配度 < 3.5 且 星标分 < 3 | 淘汰 |

### ✅ 优点

- 必须基于 README/仓库元数据的实际观察
- 至少 3 条，最多 5 条
- 具体化：写"启动时间 <100ms"而非"性能好"
- 每条注明来源（README feature list / 仓库 description / 社区反馈）

### ❌ 缺点

- 必须基于可验证来源（open issues / 长期未更新 / README 中缺失的功能）
- 至少 1 条，最多 3 条
- 如果项目确实优秀，可只说 1 条轻微不足
- 不要为了"显得客观"编造缺点

### 📖 用法

- 1-3 步，必须包含实际可运行的命令
- 安装命令 + 基础使用命令
- 复杂安装只给一条命令 + 文档链接

### ⚖️ 功能矩阵

- 行数 4-6 个核心差异化功能
- 不包含所有项目都相同的功能
- ✅ 原生 / ❌ 不支持 / ⚠️ 需插件或配置

### 🎯 场景推荐

- 🏆 最佳：综合分最高且无明显缺陷的项目
- 💎 冷门精品：stars < 100 但匹配度 ≥ 4 的项目
- 场景表：3-4 个典型使用场景，每个推荐最合适的项目
